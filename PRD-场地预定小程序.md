# PRD：运动场地预定平台

> 版本：v0.2 | 日期：2026-05-13 | 状态：待评审

---

## Problem Statement

运动爱好者（羽毛球、篮球、网球、游泳等）在寻找和预定场地时面临几个痛点：

- **信息分散**：各场馆的场地信息散布在微信群、电话、不同平台，无法一站式浏览和比价。
- **预定体验差**：传统方式需要电话确认、现场排队或通过场馆自有系统操作，流程繁琐，无法实时查看可用时段。
- **对场地方不友好**：中小型运动场馆缺乏数字化管理工具，靠 Excel 或纸质排班，容易出现重复预定、漏记、对账困难等问题。

需要一个连接 C 端运动用户和 B 端场馆的双向预定平台。

---

## Solution

一个**微信小程序 + Web 管理后台**的运动场地预定平台，采用**「先留后付」**模式：

- **C 端用户**通过微信小程序浏览附近场馆、按日期和场地号查看可用时段、一键预定占位，到场后由场馆方确认核销。
- **B 端场馆**通过 Web 管理后台发布场地信息、配置时段和价格、管理订单（确认到场/取消/核销）、查看经营数据。
- **平台方**无需预建支付链路，降低合规和开发复杂度。

---

## User Stories

### C 端用户（运动消费者）

1. 作为一个运动爱好者，我想要按**运动类型**（羽毛球/篮球/网球/游泳等）筛选场地，以便快速找到我感兴趣的运动场所。
2. 作为一个用户，我想要**按距离排序**查看附近场馆，以便优先选择离家或公司近的场地。
3. 作为一个用户，我想要在地图上直观地查看场馆分布，以便快速了解某个区域的可选场馆。
4. 作为一个用户，我想要查看场馆的**详细信息**（地址、设施照片、营业时间、联系电话），以便做出选择。
5. 作为一个用户，我想要查看某个场馆下有哪些**场地号**（如 1 号场、2 号场），以及每个场地号的类型（室内/室外、木地板/塑胶等）。
6. 作为一个用户，我想要以**日历视图**选择日期，查看当天某场馆、某场地号的**可用时段**及价格，以便直观地找到合适的时间。
7. 作为一个用户，我想要选择一个可用时段后立即**预定占位**，系统为我锁定该时段，避免被他人抢走。
8. 作为一个用户，我想要查看我的**订单列表**（待使用/已使用/已取消），以便跟踪我的预定记录。
9. 作为一个用户，我想要**取消预定**（在规定的免费取消时限内），以便灵活调整计划。
10. 作为一个用户，我想要收到预定成功、即将开始、取消等**微信服务通知**，以便及时了解订单状态。
11. 作为一个新用户，我想要通过**微信一键授权登录**，无需注册账号即可使用。
12. 作为一个用户，我想要将我常去的场馆**收藏/关注**，以便快速访问。

### B 端用户（场馆方）

13. 作为一个场馆管理员，我想要通过 Web 后台**注册并创建我的场馆**，填写名称、地址、运动类型、营业时间、联系方式等基本信息。
14. 作为一个场馆管理员，我想要上传场馆的**设施照片**，以便吸引更多用户。
15. 作为一个场馆管理员，我想要在已创建的场馆下**添加场地号**（如 1 号场、2 号场），并标注每个场地号的属性（室内/室外、地板材质、容纳人数等）。
16. 作为一个场馆管理员，我想要为每个场地号**设置可预定时段**（如 08:00-22:00，支持半小时和自定义时长）和对应的**价格**，支持不同时段不同价格（工作日/周末/黄金时段）。
17. 作为一个场馆管理员，我想要**批量设置**某一天的时段为不可预定（如闭馆日、包场活动）。
18. 作为一个场馆管理员，我想要查看本场馆的**当日订单列表**，按场地号分组展示预定情况。
19. 作为一个场馆管理员，当用户到店后，我想要**确认核销**该订单（标记为"已使用"），以完成预定流程。
20. 作为一个场馆管理员，我想要**标记用户未到场**（爽约），以便平台记录用户的信用情况。
21. 作为一个场馆管理员，我想要查看**经营数据概览**（今日/本周/本月订单数、核销率、营收统计），以便了解经营状况。
22. 作为一个场馆管理员，我想要设置**预定规则**（提前预定天数上限、免费取消时限、爽约惩罚规则）。
23. 作为一个场馆管理员，我想要在后台**手动创建预定**（如用户电话预定后由管理员代操作录入）。

### 平台运营方

24. 作为一个平台运营，我想要审核新入驻场馆的资质，确保信息真实，之后再上架。
25. 作为一个平台运营，我想要查看平台整体经营数据（场馆数量、订单量、核销率、GMV），以制定运营策略。
26. 作为一个平台运营，我想要管理运动类型的分类标签，以保持平台信息结构一致。

## Module Architecture

```
┌─────────────────────────────────────────────────┐
│                  微信云开发                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐ │
│  │ 云数据库  │ │ 云函数    │ │ 云存储（图片）    │ │
│  └──────────┘ └──────────┘ └──────────────────┘ │
└─────────────────────────────────────────────────┘
         ▲                    ▲
         │                    │
  ┌──────┴──────┐    ┌───────┴───────┐
  │  C端小程序   │    │ Web管理后台    │
  │  (微信原生)  │    │ (Next.js)     │
  └─────────────┘    └───────────────┘
```

| 模块 | 职责 | 技术 |
|------|------|------|
| 用户模块 | 微信授权登录、用户信息管理 | 云函数 + 云数据库 |
| 场馆模块 | 场馆 CRUD、场地号管理、审核流 | Web 管理后台 + 云函数 |
| 时段模块 | 时段配置、价格管理、可用性查询 | 云数据库 |
| 预定模块 | 预定锁定时段、取消、过期释放 | 云函数（事务） |
| 订单模块 | 订单生命周期、核销、爽约记录 | 云函数 + 订阅消息 |
| 通知模块 | 预定成功/取消/提醒等模板消息 | 云函数 + 微信订阅消息 |
| 统计模块 | 场馆维度和平台维度的经营数据 | 云函数 + 聚合查询 |

## Implementation Decisions

### 数据模型

**Venue（场馆）**
- `venue_id`、`name`、`address`、`latitude`、`longitude`
- `sport_type`（羽毛球/篮球/网球/游泳等）
- `phone`、`business_hours`、`photos`
- `status`（待审核/已上架/已下架）
- `owner_id`

**Court（场地号）**
- `court_id`、`venue_id`
- `name`（"1号场"）
- `type`（室内/室外）、`surface`（木地板/塑胶等）、`capacity`

**TimeSlot（时段）**
- `slot_id`、`court_id`、`date`
- `start_time`、`end_time`、`price`
- `status`（可预定/已锁定/已预定/不可用）
- `order_id`（可空）

**Order（订单）**
- `order_id`、`user_id`、`court_id`、`slot_id`
- `status`（待使用/已核销/已取消/爽约）
- `created_at`、`canceled_at`、`checked_in_at`

### 关键设计决策

1. **时段预占**：云函数事务将 `TimeSlot.status` 从「可预定」改为「已锁定」并创建订单，超时时间由场馆在后台配置（默认 15 分钟），超时未确认自动释放。

2. **预定幂等**：`TimeSlot` 唯一约束 `(court_id, date, start_time)` 天然防止重复预定。

3. **先留后付流程**：用户预定→订单「待使用」→到店后管理员核销→「已核销」。支付在线下完成，此阶段不集成微信支付。

4. **用户认证**：C 端 `wx.login` + 云函数获取 openid。B 端支持账号密码 + 手机验证码登录，以及微信扫码登录。

5. **通知**：微信小程序订阅消息，预定成功/即将开始/取消/核销时推送。

6. **Web 后台**：Next.js + 微信云开发 Web SDK，与小程序共享云数据库。

## Testing Decisions

- **云函数**：每函数单元测试，覆盖并发冲突、重复取消、权限校验。
- **小程序**：真机调试为主，核心预定流程编写 E2E 脚本。
- **关键场景**：并发争抢同时段、超时释放、跨场馆核销拦截、取消后时段复原。
- **原则**：只测外部行为，不测内部实现。

## Further Notes

- 冷启动需先引入 3-5 家核心场馆，再推 C 端。
- 地图依赖微信 `map` 组件和 `getLocation` API。
- 时段数据按日期定期归档，控制查询性能。
- 图片走云存储 CDN，配置防盗链。
